# Architecture Research: v7.0 Personal Tutor — Agno Team Integration

**Domain:** Agno Team integration into existing FastAPI + AgentOS tutoring app
**Researched:** 2026-03-15
**Confidence:** HIGH — based on direct inspection of agno 2.5.8 source code (installed at `backend/venv/`) and full v6.0 codebase

---

## Context: What Exists vs. What Changes

### v6.0 Backend Baseline

The v6.0 backend has a well-established layered architecture:

| Layer | What Exists |
|-------|-------------|
| HTTP API | FastAPI + AgentOS wrapper (`on_route_conflict="preserve_base_app"`) |
| Routers | `sessions.py`, `chat.py`, `upload.py` |
| Agents | `chat_agent`, `notes_agent`, `flashcard_agent`, `quiz_agent`, `research_agent` — all factory functions `build_X()` |
| Workflows | `build_session_workflow()` per-request factory; `run_workflow_background()` async runner |
| DB | Single `SqliteDb(id="super_tutor_traces")` instance on `app.state.traces_db`; accessed via `Depends(get_traces_db)` |
| Chat | Stateless per-request: `agent.arun(stream=True)` → `AsyncIterator` → `EventSourceResponse` |
| Session namespace | `chat:{session_id}` prefix prevents key collision between chat agent and workflow rows |

### What v7.0 Adds

A Personal Tutor tab backed by an Agno `Team`. The Team replaces the stateless `chat_agent` for the tutor tab — it coordinates specialist agents, maintains persistent multi-turn history server-side, and streams responses the same way the existing chat agent does.

The existing `/chat/stream` endpoint and stateless `ChatAgent` are UNCHANGED. The tutor is a new, separate endpoint.

## Agno Team API — Verified Facts (agno 2.5.8)

All findings below are from direct inspection of installed source at:
`backend/venv/lib/python3.14/site-packages/agno/team/`

### Constructor

```python
Team(
    members=[...],            # List[Agent | Team]
    model=...,                # Model for the coordinator LLM
    mode=TeamMode.coordinate, # coordinate | route | broadcast | tasks
    id="...",                 # stable team ID (optional, autogenerated if absent)
    session_id="...",         # default session_id (optional, can pass per-call)
    db=...,                   # SqliteDb — enables history persistence
    add_history_to_context=True,  # replay past turns from DB into context
    num_history_runs=N,       # cap on replayed turns
    instructions="...",       # coordinator system message
    stream=True,              # default streaming mode
    stream_member_events=True # propagate member agent events upstream
)
```

### TeamMode (verified from `agno/team/mode.py`)

| Mode | Behaviour |
|------|-----------|
| `coordinate` | Coordinator LLM decides which member(s) to call; synthesizes response |
| `route` | Coordinator routes to one specialist and returns that specialist's response directly |
| `broadcast` | Delegates same task to all members simultaneously |
| `tasks` | Autonomous decomposition into a shared task list; loops until complete |

For the Personal Tutor, `TeamMode.coordinate` is the correct choice: coordinator interprets the student's question and picks the right specialist (content explainer, quiz runner, flashcard generator, focus advisor).

### `arun()` vs `run()` — Streaming

`Team.arun(input, stream=True)` returns `_arun_stream(...)` which is a genuine `async def` generator function (uses `yield`). Confirmed from `_run.py` line 3211:

```python
async def _arun_stream(...) -> AsyncIterator[Union[TeamRunOutputEvent, RunOutputEvent, TeamRunOutput]]:
    ...
    # yields events in async for loop
```

**The `asyncio.to_thread()` workaround from issue #3819 is NOT needed for Team.** That issue is specific to `Workflow.run()` bypassing `save_session()` in the async path. The Team's `_arun_stream` explicitly calls `await team.asave_session(session=session)` in its cleanup path (line 3996 of `_run.py`). Session persistence works correctly in the streaming async path.

**Event type for streaming content:** `TeamRunEvent.run_content` → string value `"TeamRunContent"`. Filter pattern mirrors existing chat agent filter:

```python
# Existing chat pattern (agent):
if chunk.event == "RunContent" and chunk.content:

# New tutor pattern (team):
if chunk.event == "TeamRunContent" and chunk.content:
```

### Session Persistence

Teams use `TeamSession` (from `agno/session/__init__.py`) stored in the same `agno_sessions` SQLite table as Workflows and Agents. Session rows are keyed by `session_id`. The same `SqliteDb` instance works for all three — no separate DB needed.

**Session namespace collision prevention:** Same pattern as existing chat agent applies. Use a `tutor:{session_id}` prefix to prevent the team's session row from colliding with the workflow session row in the same `agno_sessions` table.

```python
tutor_session_id = f"tutor:{session_id}"
team.arun(message, session_id=tutor_session_id, stream=True)
```

### History Persistence

With `add_history_to_context=True` and `db=traces_db`, the Team automatically:
1. Reads past `num_history_runs` runs from `agno_sessions` on each call
2. Injects them into the coordinator's context as prior messages
3. Saves the current run to the DB at completion (via `asave_session`)

The client does NOT need to send history. This is different from the existing stateless chat endpoint where the client sends history on every request. The tutor endpoint is fully server-side stateful.

## Data Flow

### Flow 1: Tutor Message → Streaming Response

```
User types message in Personal Tutor tab
    ↓
Frontend: POST /tutor/{session_id}/stream
    body: { message: "...", tutoring_type: "advanced" }
    Accept: text/event-stream (or fetch + ReadableStream)
    ↓
tutor.py router:
    1. wf = build_session_workflow(session_id, traces_db)
       session = wf.get_session(session_id)          # reuse existing pattern
       notes = session.session_data["session_state"]["notes"]
       → 404 if session not found or notes empty
    2. team = build_tutor_team(
           session_id=session_id,
           notes=notes,
           tutoring_type=body.tutoring_type,
           db=traces_db,
       )
       tutor_session_id = f"tutor:{session_id}"
    3. async for chunk in team.arun(
           body.message,
           session_id=tutor_session_id,
           stream=True,
       ):
           if chunk.event == "TeamRunContent" and chunk.content:
               yield SSE token event
       yield SSE done event
    ↓
EventSourceResponse → client
    ↓
Frontend: appends token to in-progress message
    → on "done": message complete, render
```

### Flow 2: Tab Mount — Load History

```
User opens Personal Tutor tab (first time or after navigation)
    ↓
Frontend: GET /tutor/{session_id}/history
    ↓
tutor.py router:
    team = build_tutor_team(session_id, notes="", tutoring_type="...", db=traces_db)
    session = team.get_session(session_id=f"tutor:{session_id}")
    if not session: return []
    messages = extract_run_messages_from_session(session)
    return [{"role": ..., "content": ...}, ...]
    ↓
Frontend: renders prior conversation turns
```

### Flow 3: Session Persistence (automatic, per-request)

```
team.arun(stream=True) completes
    → agno _arun_stream() calls await team.asave_session(session=session)
    → row upserted in agno_sessions (session_id = "tutor:{session_id}")
    → next call to team.arun reads prior turns from DB automatically
       (add_history_to_context=True, num_history_runs=N)
```

## Per-Request Factory Pattern (mirroring existing chat pattern)

The existing chat router builds a new `Agent` on every request. The tutor router must do the same for `Team`. **Do not share a Team instance across requests** — `session_state` cross-contamination risk is the same as documented in `CVE-2025-64168` for Workflows.

```python
# tutor.py — per-request factory, do not store Team as module-level singleton
def _build_team_for_request(session_id, notes, tutoring_type, traces_db):
    return build_tutor_team(
        session_id=session_id,
        notes=notes,
        tutoring_type=tutoring_type,
        db=traces_db,
    )
```

The team `session_id` is NOT set at construction time — it is passed to each `arun()` call. This allows the same factory function to be used for the history endpoint without binding to a specific session.

---

## SqliteDb Sharing

The same `traces_db` instance from `app.state.traces_db` is passed to the Team via `db=traces_db`. No new SQLite file is needed. The `agno_sessions` table already holds Workflow session rows and chat Agent session rows. Team session rows are just additional rows in the same table, distinguished by `session_id` prefix.

```
agno_sessions table (existing):
  session_id = "session-workflow:abc123"    ← workflow row (session_id from Workflow)
  session_id = "chat:abc123"                ← chat agent row
  session_id = "tutor:abc123"              ← NEW: tutor Team row
```

No schema changes. No migration. The SqliteDb initializes tables on first use.

## Streaming Pattern: Team vs. Agent

| Property | Existing ChatAgent | New TutorTeam |
|----------|-------------------|---------------|
| `arun()` return type | `AsyncIterator[RunOutputEvent]` | `AsyncIterator[TeamRunOutputEvent \| RunOutputEvent]` |
| Content event name | `"RunContent"` | `"TeamRunContent"` |
| Persistence in async streaming path | Yes — Agent's `_arun_stream` saves session | Yes — Team's `_arun_stream` calls `asave_session` |
| `asyncio.to_thread()` needed? | No | No |
| Client sends history | Yes (stateless) | No (server-side history via DB) |
| Session namespace | `chat:{session_id}` | `tutor:{session_id}` |

**Critical finding:** Team.arun(stream=True) is a genuine async generator (not a coroutine). It does NOT need the `asyncio.to_thread()` workaround used for `Workflow.run()`. The streaming path in `_arun_stream` explicitly calls `await team.asave_session()` — confirmed in `_run.py` line 3996.

The SSE consumer in the router follows the same shape as the existing chat router:

```python
async def event_generator() -> AsyncGenerator[dict, None]:
    try:
        async for chunk in team.arun(
            body.message,
            session_id=tutor_session_id,
            stream=True,
        ):
            # TeamRunContent is the streaming content event
            if chunk.event == "TeamRunContent" and chunk.content:
                yield {
                    "event": "token",
                    "data": json.dumps({"token": chunk.content}),
                }
            elif chunk.event == "TeamRunError":
                yield {"event": "error", "data": json.dumps({"error": "..."})}
                return
        yield {"event": "done", "data": json.dumps({})}
    except Exception as e:
        logger.error("Tutor stream error: %s", e, exc_info=True)
        yield {"event": "error", "data": json.dumps({"error": "..."})}

return EventSourceResponse(event_generator())
```

## Member Agent Design

### Coordinator Agent (team-level LLM)

The team's coordinator model receives the student's message and decides which member to call. Its `instructions` define the routing logic:

```
You are a Personal Tutor coordinator for a study session. You have access to these specialists:
- ContentExplainer: explains concepts and answers questions from session material
- FlashcardMaker: generates new flashcard sets for specified topics
- QuizRunner: creates and evaluates quiz questions on the session material

Route each student request to the appropriate specialist. For general questions, use ContentExplainer.
```

### ContentExplainer Member

A modified `chat_agent` variant that accepts `notes` in its instructions (same grounding as the existing ChatAgent, but as a team member). The key difference from the standalone ChatAgent: it does NOT have `add_history_to_context=True` — history is managed at the Team level, not the member level.

```python
def build_tutor_chat_agent(tutoring_type: str, notes: str, db: SqliteDb | None = None) -> Agent:
    # Same persona + grounding as build_chat_agent()
    # role="ContentExplainer" so coordinator can route to it by name
    return Agent(
        name="ContentExplainer",
        role="Explains concepts and answers student questions from session material",
        model=get_model(),
        db=db,
        add_history_to_context=False,   # history managed at Team level
        instructions=f"... {notes} ...",
    )
```

### FlashcardMaker and QuizRunner Members

Reuse `build_flashcard_agent()` and `build_quiz_agent()` with an added `role=` parameter. These agents already generate JSON — the coordinator must be instructed to pass the JSON output to the frontend verbatim (not summarize it).

**Important:** When member agents are part of a Team, they do not persist their own session history — the Team's coordinator handles the thread. Member `db=` is still useful for trace capture.

## Anti-Patterns to Avoid

### Anti-Pattern 1: Using asyncio.to_thread() for Team.arun(stream=True)

**What goes wrong:** `asyncio.to_thread()` serializes execution through a thread pool. With streaming, this buffers all output until completion before returning — destroying real-time token delivery.

**Why you might do this:** The existing codebase uses `asyncio.to_thread(workflow.run, ...)` as a documented workaround for issue #3819. It is tempting to apply the same pattern to Teams.

**Why NOT to:** Verified from `agno/team/_run.py` line 3996: `_arun_stream` calls `await team.asave_session()` directly in the async path. Issue #3819 was specific to `Workflow.run()` bypassing `save_session()` in the sync path. Team has no equivalent bug.

**Prevention:** Call `team.arun(stream=True)` directly in an `async def` endpoint. No `asyncio.to_thread()`.

### Anti-Pattern 2: Sharing a Team Instance Across Requests

**What goes wrong:** Team holds internal state (`session_state`, `_cached_session`). If two concurrent requests share the same Team instance, their session states can cross-contaminate.

**Why you might do this:** Performance optimization (avoid rebuild on each request).

**Prevention:** Use the per-request factory pattern. `build_tutor_team()` is called inside each request handler and not cached. This is the same pattern used for all existing agent builders.

### Anti-Pattern 3: Storing the Tutor Session Under the Raw session_id

**What goes wrong:** The `agno_sessions` table uses `session_id` as the primary key. The Workflow already stores a row under the raw session_id. If the Team uses the same `session_id`, the Team's `asave_session()` will overwrite the Workflow's `session_data`, destroying the stored notes.

**Prevention:** Always namespace: `tutor_session_id = f"tutor:{session_id}"`. Verified pattern from existing `chat.py`: `chat_session_id = f"chat:{session_id}"`.

### Anti-Pattern 4: Setting add_history_to_context=True on Member Agents

**What goes wrong:** Member agents with `add_history_to_context=True` inject their own per-agent session history into context when called. This duplicates the team-level history and causes context bloat, confusing the specialist about what conversation it is in.

**Prevention:** Member agents in a Team should have `add_history_to_context=False` (or omit it — default is False). History is managed at the Team coordinator level via `Team(add_history_to_context=True)`.

### Anti-Pattern 5: Sending All TeamRunOutputEvent Types to the Client

**What goes wrong:** `team.arun(stream=True)` emits many event types: `TeamRunStarted`, `TeamToolCallStarted`, `TeamRunContent`, `TeamRunCompleted`, etc. Forwarding all of them as SSE requires the frontend to parse and handle a complex protocol.

**Prevention:** Filter to `TeamRunContent` for token streaming, `TeamRunError` for errors. Everything else is discarded in the event generator (same pattern as existing `chat.py` which filters to `RunContent` only).

### Anti-Pattern 6: Registering the Tutor Route Before AgentOS Wrapping

**What goes wrong:** `AgentOS(on_route_conflict="preserve_base_app")` is applied at module load time. Routes registered after wrapping may not be seen by AgentOS.

**Prevention:** All routers — including the new tutor router — must be registered with `app.include_router(...)` before `_wrap_with_agentos(app)` is called. The existing `main.py` does this correctly for all existing routers. The tutor router must follow the same placement.
